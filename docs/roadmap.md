### A.4 — FSRS-апгрейд (отложено, решение #11 из CLAUDE.md)
Добавить `difficulty: int` в `Card` (pure → 1, near-pure → 3, ровный микс → 8–10). Подробности в `docs/ssp_mmc_ru.pdf`. Полноценный переход на формулы DHP — отложен ещё дальше; не переходить без сигнала «SM-2 систематически промахивается на моих данных».

**Когда:** после A.3 и если калибровка покажет что SM-2 недостаточен.

**Анализ интервалов проведён (2026-06-16, анализ-only, прод не тронут).** Поставлен `py-fsrs` 6.3.1 (FSRS-6) в песочницу, прогон интервалов FSRS vs живой `srs.update_card`. Разбор — `docs/fsrs_analysis.md`, воспроизводимый скрипт — `tools/fsrs_interval_sim.py` (fsrs в `requirements.txt` НЕ добавлен — только dev/анализ). Главное:
- Наша SM-2-лесенка (×2.5) ≈ FSRS при retention ≈0.95 на первых шагах; дефолтный FSRS R=0.90 для нас вреден (повторы на годы — 497/1346 дней), а SM-2 множит без капа (780/1950).
- **Ключевое:** при бинарном вводе (Good/Again) Good замораживает `ease`=2.5 → pure `AA` и близкий микс `A5s` 55/45 получают ИДЕНТИЧНУЮ лесенку. FSRS с difficulty, засидированной по типу стратегии (pure→2, mid→5, близкий микс→8), показывает трудный микс ~×4 чаще на том же горизонте. Это и есть рычаг A.4 — и при бинарном вводе сидирование difficulty не опция, а необходимость (Good сам не двигает ни наш ease, ни difficulty FSRS).
- **Рекомендация (когда дойдём до A.4):** НЕ полный SSP-MMC, а «FSRS-lite на дефолтных параметрах FSRS-6» — отдельный модуль `srs_fsrs.py` параллельно SM-2 (НЕ подключённый к `srs_api`), `desired_retention≈0.95`, `maximum_interval≈120`, `learning_steps=()`, difficulty из уже существующего `Card.classify()`. Параметры пока НЕ фитить (данных нет — 15 ответов на 16.06; бинарный ввод обедняет обучение difficulty). A/B — после 2–3 недель данных (A.2). Согласовано с #11.
- **Методика первоисточника vs SM-2 (концептуально, 2026-06-16):** разобрана в `docs/fsrs_analysis.md` §0 (по самой статье `3534678.3539081.pdf` в корне, не по пересказу). Суть: у статьи есть обученная модель памяти (DHP) + оптимизация стоимости (SSP-MMC), у нас ни того ни другого. Три значимых отличия — нет модели памяти (просто множим интервал); мёртвая ось трудности при бинарном вводе (Good не двигает ease → `AA` и микс 55/45 одинаковы); нет цели-удержания/потолка. Важно: **статья (Appendix A.2) сама предписывает SM-2 как фазу холодного старта** → наш SM-2 не ошибка, а ровно стартовая фаза по статье. Полный SSP-MMC не наш масштаб (220 млн повторов vs 15 ответов; миллионы юзеров vs 1 игрок); переносимое = живая трудность (сидируется из стратегии) + retention≈0.95 / cap≈120 → носитель FSRS-lite (см. рекомендацию выше).
- Тесты двигателя зелёные (не трогали): `test_srs` 64 + `test_srs_api` 18.

**FSRS-lite каркас собран (2026-06-17) — §5 анализа реализована, прод/`srs_api` не тронуты.** «Следующий разумный шаг» из `fsrs_analysis.md` §5 теперь в коде (вариант B — отдельный модуль, не подключённый к API):
- **`srs_fsrs.py`** — планировщик поверх `py-fsrs` (FSRS-6), **параллельно** SM-2. Конфиг: `desired_retention=0.95`, `maximum_interval=120`, `learning_steps=()`. `seed_difficulty(strategy)` сидирует difficulty из частот стратегии (pure→2.0, near-pure(0.9)→~4.4, ровный микс→8.0, мульти-вей→clamp 10; пусто→5.0; ключ = `max(freq)`, симметрично вокруг 0.5). `replay()` — по реальной истории (delta-driven); `good_ladder()` — проекция all-Good (due-date-driven). Импорт `fsrs` ленивый → `import srs_fsrs` и `seed_difficulty` работают без пакета; `fsrs` НЕ в `requirements.txt` (dev-only — прод модуль не импортит).
- **`test_srs_fsrs.py`** — 45 проверок. seed-якоря/монотонность/границы/симметрия гоняются всегда; конфиг/лесенка-кап/дифференциация трудности/lapse/детерминизм — при наличии fsrs, иначе аккуратный SKIP. Зелёные.
- **`tools/fsrs_ab_compare.py`** — A/B-харнесс: для каждого дека `srs_state/*.srs.json` гоняет реальную историю карты через живой `srs.update_card` и FSRS-lite, печатает spread сидов трудности (по всему деку) + сравнение по отвеченным картам + forward-проекцию all-Good. Сам `srs_state` не пишет.
- **Что видно на реальных данных (честно — тонко):** на 15 (GTOWNL10) и 10 (NL25GTOW) ответах за 1 день интервалы почти не разошлись (по 1 повтору/карту), НО дифференциация трудности уже читается: микс `66 UTG (open 0.27/fold 0.73)` сидируется D≈7.1, pure-руки — D=2.0, а forward-проекция показывает то самое — SM-2 даёт `AA` и `66` ИДЕНТИЧНУЮ лесенку (`1 3 8 20 50 125…`, ease заморожен), FSRS-lite — `66` ~2× чаще. Харнесс даже поймал реальные lapse (`J7o`/`65s` отвечены Again → ease 2.30 / низкая стабильность).
- Регрессий нет: `test_srs` 64 + `test_srs_api` 18 + `test_srs_fsrs` 45 зелёные.

**Решения пользователя:**
- **Уведомления — только баннер в аппе** (не email/push). При заходе «ты сегодня не тренировался»; `last_active` берётся из журнала (есть ли строка за сегодня). Ноль доп-инфраструктуры; email/push можно прикрутить позже поверх того же `last_active`.
- **Лимит карт на споте+позиции — дневная ЦЕЛЬ, не блокирует.** Таблица `goals` (per-user: target N на mode/spot/position) + прогресс = счёт сегодняшних строк журнала; очередь не режем. *Нюанс для схемы:* в Drill цель = объём практики (генерит сколько угодно), в Learn ближайший аналог = «новых карт/день на спот» (повторы планирует алгоритм) → цель делаем mode-aware; для Learn это дробит сегодняшний deck-wide `new_limit` в `get_due_cards` на per-spot.

### Этап 3 — per-action EV и EV отрицательных рук *(идея, не начат)*

**Что сейчас.** Парсер берёт одно число из `.rtc_value` клетки — EV стратегии в точке безразличия. Для чистых рук это EV единственного действия. Для миксов оба действия равнозначны (GTO-индифферентность) — одно число корректно. Отрицательные EV хранятся в `evMap`, но в Drill пилюля скрыта (`ev < 0 → None → нет пилюли`).

**Что хочется добавить:**

1. **Per-action EV.** В правой панели GTOW для каждой руки видны отдельные строки: `Allin 100 → -6.19`, `Raise 21.5 → 0.01`, `Call → 0.01`, `Fold → 0`. Для чистых рук все три числа разные. Это даёт понять *почему* алл-ин с KQs vs 3-бет плохой, а не просто «ошибка». Схема хранения: `ev.vs_3bet.MP.vs_CO.KQs` сейчас `float` → расширить до словаря с порядком как в GTOW: **сначала allin, затем рейзы по убыванию размера, предпоследнее call, последнее fold**. Пример: `{"allin": -6.19, "4bet": 0.01, "call": 0.01, "fold": 0}`. Порядок важен для отображения в Drill — пользователь привыкает видеть одну и ту же последовательность кнопок что в тренажёре, что в солвере. Обратно совместимо: старый `float` интерпретируется как EV лучшего действия.

2. **EV отрицательных рук.** Сейчас K7s с EV -0.01 хранится, но не показывается. Имеет смысл показывать «ты сыграл в стратегии, EV этого хода −0.01 BB — ты на краю безразличия». Честнее и информативнее.

**Что нужно для реализации:**
- `gtow_parse.py`: для каждой руки кликать по ней и читать per-action строки из правой панели. Это 169 кликов на матрицу — реально в автоматическом режиме через Claude in Chrome, но в ~10 раз медленнее.
- Альтернатива: изменить вид матрицы через GTOW-переключатель, если там есть экспорт per-action (не найдено пока).
- `range_engine.get_ev()`: вернуть dict вместо float.
- `drill_engine`/`drill.js`: показывать «EV этого действия: X BB» на feedback-панели.

## Форматы стола — heads-up (2-max) + 3-max (спины) *(идея, не начато — 2026-06-17)*

> **→ heads-up @ 100bb консолидирован в Track E** (`docs/multiformat.md`, этап E1). **Здесь остаётся «спин» = 3-max + короткий стек + анте** как отдельная будущая связка (три оси разом, см. ниже) — она вне ветки мультиформата.

**Заметки на будущее (design-first, ничего не решаем сейчас):**

- **«Спин» ≠ просто меньше игроков — это три оси разом.** Спин = 2/3 игрока **+ короткий стек** (обычно 25bb, гиперы 15bb и меньше) **+ анте** + winner-take-most. Формат стола добавляется дёшево (см. ниже), но «настоящий спин» цепляет ещё отложенную ось глубины стека (секция выше) и анте, которых сейчас нет. Эти три вещи проектировать **вместе** — иначе выйдет HU/3-max на 100bb без анте, что спином не является. Это главная заметка.
- **Диапазоны = новые паки, в идеале ноль правок движков.** По прецеденту `meta`/per-`<stem>` (как с глубиной): формат уходит в `meta.table_size` (`2max`/`3max`) + `config.positions`, один пак = один формат+глубина; SRS-стейт и статистика уже per-pack. Споты сужаются: HU — RFI (SB=BTN) + vs_RFI (BB); 3-max — BTN/SB/BB. `config.positions`/`rfi_positions`/`vs_rfi_options` это уже параметризуют per-pack → движки, вероятно, не трогаем (проверить на конкретном паке).
- **Рассадка (Track B, дёшево).** `SLOT_POS` сейчас 6/8/9-max, компонент n-агностичен (заметка B.1). Нужны координаты 2-max (hero внизу, villain сверху) и 3-max (треугольник). Чистый CSS/JS, без движка.
- **Источник данных — открытый вопрос.** Есть ли в GTOW спин-солвы (HU/3-max, короткий стек, анте)? Проверить когда дойдём. Для коротких стеков альтернатива — push/fold чарты: наш бинарный грейдинг на них ложится хорошо (allin/fold).
- **Анте** пока не моделируется (префлоп — теоретические no-ante диапазоны, решение #2 в CLAUDE.md). Для спинов анте принципиально (меняет диапазоны) — отдельный кусок.

**Приоритет:** не задан (идея на будущее). Браться отдельным треком, design-first, когда вернёмся к расширению форматов.

## Префлоп — мультивей / cold-calls: расширение схемы *(новый трек, решение 2026-06-14)*

**Откуда.** Структурный анализ дерева GTOW (чат 2026-06-14): сверяли наше разбиение `(spot, hero, villain)` с форматом визарда.

**Что нашли.**
- Дерево GTOW — полное последовательное (UTG→…→BB→FLOP), узел = место + история действий; в узле Fold / Call / Raise(size) / Allin, размеры контекстные (опен 2.5; 3-бет 7.5 vs UTG, 10 vs поздних; **сквиз 14**, если впереди есть коллер).
- Ключевая ось в библиотеке решений — **Preflop bet sizes**: `No cold calls` (Simple-дерево: непустые позиции против опена только 3-бет/фолд, колл только у BB закрывающим действием → **всё heads-up**) vs `With cold calls` (флэт разрешён → **мультивей + сквизы**). Текущий пак `NL25GTOW` — `With cold calls`.
- Наша heads-up-схема (один вилан) ложится **1:1 на No-cold-calls** дерево, но **не покрывает** cold-call-узлы: сквиз = опен + коллер + hero, а поле `villain` одно.
- Парсер на cold-call-узле молча мислейблит (см. `problems.md` P-026).

**Решение пользователя (2026-06-14):** **расширять схему под мультивей** (не ограничиваться No-cold-calls и не просто чинить парсер). Самый крупный путь → ведём строго **design-first**, по этапам, не делаем несколько рискованных вещей разом.

**Staged-план (Этап 0 done 2026-06-14 — формат зафиксирован):**
- **Этап 0 — дизайн модели данных** *(done 2026-06-14)*. **Выбран вариант (B): новый спот `squeeze`** (расширенный кортеж), НЕ action-path (A). Обоснование: наименьший шаг, покрывающий доминирующий мультивей-узел; зеркалит инфру `vs_RFI`; сайзы остаются абстрактными (в коде, не в ключе); миграции текущих 3 спотов нет; мульти-коллер достаётся бесплатно (длиннее `+`-список). Честный размен: (B) менее общий — экзотика (`vs_squeeze` = опенер против сквиза, `overcall`, сквиз-над-3бетом) откладывается, каждый станет отдельным спотом по надобности; (A) может зайти отдельным лосслесс-представлением в анализатор раздач (Milestone 3), если понадобится полное дерево — это не ломает (B).

  **Зафиксированный формат squeeze (Этап 0):**
  - Спот `squeeze`; под-ключ `"<opener>-<caller>"` в порядке посадки — опенер первым, коллер вторым (коллер не действует раньше опенера → порядок всегда однозначен и обратим). Разделитель `-` (НЕ `+`: позиция может содержать `+`, напр. `UTG+1` → split сломался бы). Мульти-коллер позже: `opener-c1-c2` (первый = единственный рейзер, остальные — коллеры).
  - Инвариант спота: **ровно 1 рейзер (опенер) + ≥1 коллер + hero.**
  - Действия: `squeeze`, `call`, implicit `fold` (`Raise`+`Allin` → `squeeze`, как Raise+Allin→`4bet` в vs_3bet, решение #13).
  - EV: `ev.squeeze[hero]["<opener>-<caller>"][hand] = float` (зеркально `spots`).
  - Config: `squeeze_options: { "<hero>": ["BTN-SB", ...] }` (как `vs_rfi_options`).
  - card_id: `"AKs__BB__squeeze__BTN-SB"` (vs-форма, villain = пара; режется по `__`, внутри пары `-`).
