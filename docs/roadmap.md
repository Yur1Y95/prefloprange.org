## Drill — GTO-EV прибыльного действия

Реальный посчитанный GTO-EV открытия вместо синтетических констант. Решение #13 в CLAUDE.md, формат блока `ev` — в «Формат range-файлов».

### Этап 1 — код (done 2026-05-30)

Синтетика (`EV_CORRECT_OPEN` и т.д.) с правильного пути RFI убрана. Тесты `test_drill_ev.py` — 28 зелёных; смоук на реальном `cash_micro_100bb.json` (без блока `ev`) — graceful fallback, не падает.

### Регресс EV (был обнаружен 2026-05-31, ВОССТАНОВЛЕНО в тот же день)
Блок `ev` в `NL25GTOW.json` отсутствовал (`grep -c '"ev"'` → 0, и в git HEAD 0) — данные Этапа 2 либо не дошли до файла, либо затёрты пересохранением в Editor (`_edBuildRangeData` не пишет `ev`). **Восстановлено 2026-05-31:** пользователь прислал 5 скриншотов GTOW (UTG/MP/CO/BTN/SB), числа сняты заново и записаны в `ev.RFI` скриптом со структурной проверкой (`build_ev.py` в scratchpad). ### Этап 2 — импортёр / ручной ввод EV для RFI *(данные восстановлены 2026-05-31, workflow ниже валиден)*
Первый пакет — **GTOW NL25** (`data/NL25GTOW.json`). Источник EV — PNG-скриншоты матрицы GTOW (вкладка «стратегия + EV»): число в правом нижнем углу каждой клетки. Один скриншот = одна позиция RFI.

**Сделано:** блок `ev.RFI` заполнен для **UTG, MP, CO, BTN, SB** ручным визуальным съёмом со скриншотов с пошаговой сверкой пользователем. Итог по числу записей = числу играбельных рук: UTG 47, MP 55, CO 66, BTN 88, SB 106. Красные клетки с EV 0.00 сохранены как `0.0` (реальный маргинальный опен). Проверено программно: каждая рука с `open>0` или `call>0` в `spots.RFI` имеет ровно одну EV-запись, лишних/пропущенных нет; `range_engine.get_ev` отдаёт числа, на фолд-руках `None`.

**Решения, зафиксированные по ходу:**
- **SB-миксы:** в смешанной стратегии (open + call BB) EV открытия и колла **равны** (условие индифферентности GTO) → храним одно число на руку, без привязки к действию. Это совпало с моделью `get_ev` (один `float` на руку). Call-only руки SB (74s, 87o, 97o, J8o, Q7o, T4s — все EV 0.0) тоже получили запись: их прибыльное действие = call.
- **K3s MP** open 0.15 / EV 0.0 — подтверждено пользователем как корректное (не опечатка диапазона).
- Поправки сверки: CO 76s и 65s = 0.01 (а не 0).

**Workflow (отработан, переиспользуем для будущих пакетов):** Claude снимает числа со скриншота глазами → показывает таблицей на сверку → только после ОК пишет. Визуальный съём + ручная сверка надёжнее OCR на мелком шрифте. В файл пишем одной правкой после сверки всех позиций.

**Открытое:** EV только для RFI; vs_RFI/vs_3bet в `NL25GTOW.json` пусты (структура в `ev` готова, чисел нет). На vs-спотах в Drill остаётся старая синтетика (см. решение #13). ## Префлоп — мультивей / cold-calls: расширение схемы *(новый трек, решение 2026-06-14)*

**Откуда.** Структурный анализ дерева GTOW (чат 2026-06-14): сверяли наше разбиение `(spot, hero, villain)` с форматом визарда.

**Что нашли.**
- Дерево GTOW — полное последовательное (UTG→…→BB→FLOP), узел = место + история действий; в узле Fold / Call / Raise(size) / Allin, размеры контекстные (опен 2.5; 3-бет 7.5 vs UTG, 10 vs поздних; **сквиз 14**, если впереди есть коллер).
- Ключевая ось в библиотеке решений — **Preflop bet sizes**: `No cold calls` (Simple-дерево: непустые позиции против опена только 3-бет/фолд, колл только у BB закрывающим действием → **всё heads-up**) vs `With cold calls` (флэт разрешён → **мультивей + сквизы**). Текущий пак `NL25GTOW` — `With cold calls`.
- Наша heads-up-схема (один вилан) ложится **1:1 на No-cold-calls** дерево, но **не покрывает** cold-call-узлы: сквиз = опен + коллер + hero, а поле `villain` одно.
- Парсер на cold-call-узле молча мислейблит (см. `problems.md` P-026).

**Решение пользователя (2026-06-14):** **расширять схему под мультивей** (не ограничиваться No-cold-calls и не просто чинить парсер). Самый крупный путь → ведём строго **design-first**, по этапам, не делаем несколько рискованных вещей разом.
