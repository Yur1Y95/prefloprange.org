## 0. Зачем это и что решаем

Задача чата (2026-06-19): добавить **план тренировки**, **дашборд** (прогресс за неделю/месяц, стрики) и **честное определение тренировки** (анти-фарм). По решению пользователя строим это не на файлах, а сразу на **Postgres/Supabase** — то есть начинаем Track D, Этап 0 (миграция хранилища). Этот чат — **только проектирование схемы** (дизайн до кода); поднятие Supabase и код API — следующие чаты.

Почему всё это упирается в схему: и план, и стрики, и календарь — это вопросы вида «что и когда пользователь сделал». Ответить на них можно только если каждый ответ записан **с датой**. Сейчас:

- **Drill** пишет агрегаты (`stats.json`) и `history.json`, где `ts` — это `"ЧЧ:ММ:СС"` **без даты**, скользящие 200 записей. По дням не восстановить.
- **Learn (SRS)** уже пишет по-ответную историю с датой (`card.history` = `{date, rating, correct, delta_days}`), но разбросанную по `srs_state/*.json` и без сводки по дням.

Вывод: нужна одна таблица-журнал, куда пишут оба тренажёра.

## 1. Главная идея — журнал событий (event log)

Центр схемы — **одна append-only таблица `answers`**: одна строка на каждый отвеченный спот, из любого тренажёра. Её **только дополняют**, не правят и не удаляют.

Всё остальное — это **запросы поверх журнала**:

- статистика по спотам/позициям (бывший `stats.json`) → `GROUP BY spot, position`;
- лента истории (бывший `history.json`) → `ORDER BY ts DESC LIMIT n`;
- дневная активность, точность, объём → `GROUP BY день`;
- стрики, «тренировался ли каждый день» → запрос над днями;
- прогресс целей → счёт строк, подходящих под цель.

Почему так, а не «таблица на каждую метрику»: метрики меняются и множатся (завтра захочется «точность по мастям» или «утренние сессии»), а сырьё — всегда одно и то же событие «ответил на руку». Если хранить **факты** (что произошло), любую метрику можно посчитать потом, в том числе ту, которую ещё не придумали. Если хранить только агрегаты — придётся переписывать запись каждый раз. Это классический приём (event sourcing в миниатюре).

Отдельно от журнала живёт таблица `cards` — **текущее состояние SRS** (что показать следующим). Разделение принципиальное:

> **`answers` = «что случилось» (история, неизменна). `cards` = «что показать дальше» (состояние, меняется).**

## 2. Таблицы

Полный DDL — в `db/schema.sql`. Здесь — назначение и «почему» по каждой.

### 2.1. `profiles` — пользователь (поверх Supabase Auth)

Аутентификацию (логин, email, Google-OAuth) держит сам Supabase в схеме `auth`. Мы заводим свою `profiles`, привязанную к `auth.users.id`, под **прикладные** поля:

| Колонка | Зачем |
|---|---|
| `id uuid` | = id пользователя из Supabase Auth (в Supabase добавить `references auth.users(id)`) |
| `timezone text` | **Часовой пояс пользователя.** «День» для стриков считается по его локальному дню, не по UTC (см. §3.2) |
| `session_min_hands int` | **Порог «честной тренировки»** (анти-фарм). День засчитан, только если рук ≥ этого числа. Дефолт 20, настраивается |
| `created_at` | служебное |

### 2.2. `answers` — журнал событий (сердце схемы)

Одна строка = один отвеченный спот. Заменяет **и** `stats.json`, **и** `history.json`.

| Колонка | Зачем |
|---|---|
| `id bigint` | первичный ключ |
| `user_id uuid` | чей ответ |
| `ts timestamptz` | **полная метка времени в UTC** — основа всех «по дням» |
| `mode text` | `drill` / `learn` — оба тренажёра пишут в один журнал |
| `pack text` | какой range-пак (`GTOWNL10` и т.д.) |
| `spot, position, villain` | идентичность спота (как в range-файлах; `villain` пуст для RFI) |
| `hand text` | рука в нотации (`AKs`) |
| `action text` | что выбрал пользователь |
| `correct bool` | попал в стратегию или нет |
| `is_timeout, revealed bool` | таймаут; в Learn — нажал «Показать ответ» |
| `rating smallint` | SM-2/FSRS-оценка 1–4 (Learn); `NULL` для Drill |
| `ev real` | GTO-EV действия в bb, если известен |
| `correct_action text` | что хотела стратегия (для разбора в UI) |
| `card_id text` | ссылка на SRS-карту (Learn), формат как `srs.Card.card_id` |

Ключевое поле для будущего — **`rating`**: логируя оценку каждого ответа, мы копим ровно те данные, на которых потом фитится FSRS (Track A.4). Без журнала их негде взять.

### 2.3. `cards` — состояние SRS (per-user)

То же, что сейчас `srs_state/<pack>.srs.json`, но в таблице и по каждому пользователю. Хранит **только текущее** расписание.

Важный момент миграции: в JSON у карты был массив `history[]`. **В Postgres мы его НЕ дублируем** — эти события переезжают в `answers`. История карты теперь = `SELECT * FROM answers WHERE card_id = ... ORDER BY ts`. Ничего не теряется, зато история едина с Drill и доступна для аналитики.

Поля SM-2 (`ease_factor`, `interval_days`, `next_review`, `last_seen`, счётчики) — один в один из `srs.Card`. Добавлены **nullable** `stability` и `difficulty` под FSRS-6 — чтобы выкатка A.4 **не требовала миграции** (просто начнём их заполнять).

### 2.4. `goals` — гибкий «план тренировки»

Пользователь уточнил: «100 открытий на позицию» — это пример, а не жёсткая модель. Поэтому цель — это **цель на (mode, опциональный охват) за период**:

| Колонка | Зачем |
|---|---|
| `mode` | `drill` / `learn` |
| `pack, spot, position, villain` | охват; `NULL` = «любой/в сумме» |
| `period` | `daily` (сбрасывается каждый день, кормит стрик) или `total` (накопительный курс/чек-лист) |
| `metric` | `hands` (ответы) / `correct` (верные) / `new_cards` (новые карты — рычаг для Learn) |
| `target int` | сколько |

Так выразимы обе трактовки и многое сверх: «Drill / RFI / UTG / 100 в день», «Drill / любой спот / 300 в день», «Learn / vs_3bet / 20 новых карт в день», «Drill / vs_RFI / всего 2000» (курс). Пример «100 опенов на каждую позицию RFI» = пять строк (или одна с `position=NULL` и target 100, если считать в сумме).

**Mode-aware — это про FSRS.** В Drill цель — объём практики (генерим сколько угодно руки). В Learn нельзя «ответить N повторов» — сколько карт созрело, столько алгоритм и даст. Поэтому дневная цель Learn измеряется в **новых картах/день** (`metric='new_cards'`) — единственном рычаге, которым там реально управляют (это дробит сегодняшний deck-wide `new_limit` из `get_due_cards` на per-spot). Повторы остаются на усмотрение планировщика — и SM-2 сейчас, и FSRS потом.

## 3. Ключевые решения дизайна

### 3.1. История выносится из `cards` в `answers`
Один источник правды по событиям. История карты восстанавливается запросом; FSRS-фит = выборка по `card_id`. Verified: см. §5 («sample card → N journal rows»).

### 3.2. «День» — это локальный день пользователя (ловушка timezone)
Стрики и «тренировался сегодня?» зависят от того, что считать «сегодня». Если бакетить по UTC, игрок в Москве, занимавшийся в 01:00, попадёт во «вчера». Поэтому: **`ts` храним в UTC** (`timestamptz`), а в день переводим через `(ts at time zone profiles.timezone)::date`. Это не косметика — от этого зависит, не «порвётся» ли стрик на ровном месте.

### 3.3. «Честная тренировка» = день с ≥ `session_min_hands` рук (анти-фарм)
Прямой ответ на «не фарм ради красивой картинки». Одна рука не «зажигает» день. Реализовано во вьюхе `v_session_days` (`HAVING count(*) >= session_min_hands`), и **и календарь, и стрики строятся только из квалифицированных дней**. Порог в `profiles`, дефолт 20, настраивается. (Считаем суммарно по обоим режимам за день — «позанимался» засчитывается любой практикой.)

### 3.4. FSRS-готовность заранее
`answers.rating` логируется с первого дня; `cards.stability/difficulty` уже есть (nullable). Переход на FSRS-lite (A.4) — без миграции схемы.

## 4. Как три фичи ложатся на схему

| Фича | Где живёт | Как считается |
|---|---|---|
| **План тренировки** | `goals` + журнал | строка(и) цели; прогресс = счёт сегодняшних (`daily`) или всех (`total`) строк `answers` под охват цели. Learn-дневная — через `new_cards`. Запросы 1–2 в `queries.sql` |
| **Честная сессия (анти-фарм)** | `profiles.session_min_hands` + `v_session_days` | день засчитан только при ≥ N рук; стрики/«trained» в календаре — только по таким дням |
| **Дашборд неделя/месяц** | `v_daily_activity`, `v_session_days`, `v_streaks` | календарь-хитмап (запрос 3: каждый день окна + флаг `trained`), бары объёма/точности по дням и неделям (запрос 4), счётчики текущий/лучший стрик (запрос 5) |

«Календарь + бары» (выбор пользователя) кормятся из одних и тех же вьюх — добавить второй вид графика дёшево.

## 5. Что показала верификация

`tools/verify_schema_sqlite.py` (sqlite, т.к. в песочнице нет Postgres/PyPI; sqlite доказывает **логику** запросов, авторитетный DDL — Postgres). Прогон 2026-06-19 — **все проверки зелёные**:

**Часть A — реальные данные** (через `tools/migrate_to_pg.py`): 1859 `cards` + 225 `answers` (25 learn из `card.history`, 200 drill из `history.json`) загрузились в схему; дневной агрегат сошёлся с независимым пересчётом на Python; у всех 25 learn-ответов есть `rating`; история карты восстановима из журнала.

**Часть B — синтетика (многодневная)**: анти-фарм отбросил день с 5 руками; longest streak = 3; current streak = 1 (после разрыва) и = 3 (если сегодня ещё не тренировался — считается по вчера); календарь покрыл все дни окна с верными флагами `trained` [0,1,1,1,0,1]; дневная цель 25/20 (выполнена), `total` = 105; `new_cards` за день = 1.

## 6. Миграция текущих данных

**Обновление 2026-06-22 (D.1-api-cards): для `cards` выбран ЧИСТЫЙ СТАРТ — миграция НЕ применялась** (решение пользователя). При первом включении БД колода Learn строится заново как полностью новая; накопленные `srs_state/*.json`-интервалы остаются на диске, но в `cards` не переносятся. `migrate_to_pg.py` остаётся как готовый экстрактор на случай, если позже захочется перенести историю `answers` (learn-события из `card.history`), но для состояния `cards` он сознательно не задействован. Ниже — что бы перенеслось, если бы мигрировали (оставлено как справка):

- **Learn**: `card.history` → строки `answers` (mode=learn) с датой и оценкой — переезжает чисто. `action` в истории не хранился → `NULL` (есть `correct`+`rating` — этого хватает дашборду; `correct_action` берём из доминирующего действия стратегии).
- **Drill**: `history.json` без даты → все строки штампуются датой mtime файла (видно в прогоне: 200 drill-строк сели на один день `2026-06-18`). И **нет поля `pack`** → `(unknown)`. Это и есть цена «Drill раньше не журналировал даты»; с переходом — пишем полную метку и пак сразу.
- `stats.json` отдельно не мигрируем — он производный, восстановится как `v_stats` из журнала (хотя исторические агрегаты до журнала точными по дням не станут — данных нет).
