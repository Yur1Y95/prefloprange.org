## P-001: Edit → Save в Editor'е не доезжает до Visualizer'а

**Status:** resolved
**Found:** 2026-05-28
**Resolved:** 2026-05-28
**Severity:** high — блокирует основной workflow редактирования диапазонов

### Симптомы
1. Открываешь диапазон в Editor, правишь матрицу, жмёшь Save → визуально OK («✓ Saved!»).
2. Переключаешься в Visualizer → видишь **старую** версию без правок.
3. В списке файлов Editor'а появляются **два** одинаковых имени, например «Cash 6max 100bb (GTO micro)» и «Cash 6max 100bb (GTO micro) ★».

### Корень
- `_edDoSave` в `static/editor.js` (строка 627) пишет в **браузерный `localStorage`** через `UserStorage.save()` из `static/storage.js`.
- На сервере **нет POST-эндпоинта** для записи range-файлов. Только GET'ы `/api/ranges/list` и `/api/ranges`.
- Visualizer (и Drill, и Editor-load-dropdown) мержат список из двух источников: `await fetch('/api/ranges/list')` (сервер) + `UserStorage.allFileEntries()` (localStorage). Поэтому когда юзер сохранил локально — в списке появляется второй вариант с `★` (этот суффикс добавляется в `UserStorage.toFileEntry()`).
- Визуализатор по умолчанию открывает server-версию (без `★`), которая физически не менялась — отсюда «правки потеряны».

### Решение (выполнено)
1. **Бэкенд.** Добавлен `POST /api/ranges/save` в `main.py`. Тело: `{filename, data}`. Защита от path-traversal через regex `^[A-Za-z0-9_\-]+$` на basename. Перезаписывает существующий файл без вопросов (single-user workflow). Покрыто файлом `test_ranges_api.py` — 7 тестов: roundtrip save/list/read, overwrite, нормализация имени без `.json`, 8 крафтовых path-traversal входов, пустое имя, не-dict в `data`.
2. **Editor.** `_edDoSave` стал async — шлёт `POST /api/ranges/save` вместо `UserStorage.save`. На ошибке показывает сообщение из `detail`. Кнопка `⬇ Export` теперь делает чистый browser-download (Blob → URL → `<a download>`), не трогает localStorage. Кнопка `⬆ Import` удалена вместе с её обработчиком и `<input id="eImportInput">` в HTML.
3. **`UserStorage`-merge удалён** из всех потребителей: `editor.js` (dropdown `My saved ranges` оптгруппа), `visualizer.js` (boot, `vizRefreshAfterSave`, `vizLoadFile`), `drill.js` (boot, `drillLoadConfig`, `drillRefreshFileList`). Везде сервер теперь единственный источник.
4. **`storage.js`** перезаписан в tombstone-стаб (sandbox не дал удалить файл). Тэг `<script src="/static/storage.js">` из `index.html` снят.
5. **Cache-bust** поднят на edited JS: `drill.js`, `visualizer.js`, `editor.js` → `?v=20260528a`.

Итого: 51 + 17 + 7 = **79 тестов зелёные** (4 новых от ranges-api).

### Регрессии
*(пока не обнаружены)* — нужна ручная верификация на реальном workflow: edit → save → switch to Visualizer → проверить что правки видны. Также проверить что Drill корректно загружает диапазон после правки (он использует `drillLoadConfig` для подгрузки спотов).

**Известные риски, мимо которых прошли:**
- Старые записи в `localStorage` пользователей не очищаются автоматически — если у тебя там накопились ★-диапазоны, они останутся жить в браузерном хранилище как мусор. Очистить вручную можно через DevTools → Application → Local Storage → удалить ключи с префиксом `range:`.
- `drill2.js` остался с UserStorage-кодом, но не подключён в `index.html` — мёртвый файл, к которому никто не обращается. Удалить как уборку.

## P-002: Интерфейс не оптимизирован под мобильные устройства

**Status:** resolved (для Drill/Learn/Visualizer; Editor/Postflop — частично)
**Found:** 2026-05-28
**Resolved:** 2026-05-29
**Severity:** medium — десктоп-only сейчас, мобильный workflow невозможен

### Симптомы
- На viewport 375–414px (iPhone) layout-grid `190px / 1fr / 200px` ломается: sidebars наезжают на центр или обрезаются.
- Стол `.table-wrap` фиксированных 560×360px (теперь 680×330) не помещается на типичном телефоне.
- Editor-матрица 13×13 нечитаема на маленьком экране.
- Action-кнопки могут быть меньше touch-friendly минимума (44×44pt по Apple HIG / 48dp по Material).
- Топ-навбар с 6 табами обрезается на узком viewport.

### Корень
CSS написан жёстко под десктоп. Нет ни одного `@media` правила в `static/style.css`. Размеры стола, матриц, sidebar'ов — пиксельные константы.

### Решение (выполнено 2026-05-29, Track B.4 Этапы 1–3)
Единый блок `@media (max-width:768px)` в конце `style.css` (только ADD-оверрайды, десктоп нетронут) + новый изолированный `static/mobile.js`. Полное описание этапов и решений — `roadmap.md` → Track B.4. Кратко:
- Табы скроллятся горизонтально (остались сверху по решению пользователя); layout → single column, стол наверху, статистика стекается под ним.
- Стол резиновый (`min(94vw,680px)`+`aspect-ratio`), рассадка в `%` едет сама.
- Левая панель настроек → **bottom-sheet, открывается свайпом вверх** (паттерн выбран пользователем поверх изначального «боковой drawer»).
- Кнопка Deal Hand вынесена в **плавающий FAB** (она физически внутри панели → пряталась в лист).
- Touch-таргеты ≥44px.

**Связанные фиксы по ходу:**
- Баг позиционирования фишек блайндов: `renderChips` в `drill.js` считал от фикс. `W=680,H=330` → фишки улетали за экран на уменьшенном столе. Исправлено на реальные размеры стола + пропорциональный скейл (десктоп: множитель 1, без изменений).
- `position:fixed` у потомка трансформированного `.panel-left` позиционируется относительно предка, не вьюпорта → FAB и backdrop вынесены на `body`.

**Проверено:** headless Playwright на iPhone SE/15, Galaxy S20 + десктоп-регресс. Нет переполнения, лист и FAB работают, десктоп идентичен прежнему.

### Регрессии
Не обнаружено (десктоп проверен harness'ом: `.panel-left` остаётся `position:static`, ручка и FAB `display:none`). **Не покрыто:** Learn card-view и Editor-матрица на мобилке (см. «Открытое» в roadmap B.4) — отдельный заход. Нужна ручная проверка на реальном телефоне (ngrok).

## P-006: Villain Range — дро накладываются на made-hands, сумма 113.5%

**Status:** open
**Found:** 2026-05-29

## P-007: Матрица — зелёный «хвост» (1px) у правого края ячеек со смешанной частотой

**Status:** resolved
**Found:** 2026-05-29
**Resolved:** 2026-05-29
**Severity:** low — косметика, но мешает читать матрицу (Track B.3)

### Симптомы
В матрице 13×13 (Visualizer и Editor) ячейка с частичной частотой open показывает зелёную полоску заливки = частоту И тонкую вертикальную полоску у самого правого края. Пользователь: «зелёный хвост в конце заливки видно».

### Корень (настоящий — подтверждён пиксельным пробингом в реальном Chromium)
Заливка ячейки — CSS `linear-gradient(to right, …)` поверх `.matrix-cell` с `border:1px solid rgba(255,255,255,.06)` + `border-radius:3px` + `overflow:hidden`. При **`background-clip: border-box`** (дефолт) градиент рисуется и под бордером; на правом краю Chromium оставляет 1px стартового цвета градиента (зелёного) под полупрозрачным бордером. Композит даёт самый правый пиксель `rgb(58,132,62)` = `rgba(255,255,255,.06)` поверх зелёного `#2E7D32(46,125,50)` — яркая зелёная вертикальная линия в 1px. **Это артефакт рендеринга края, не fold-остаток.**

Диагностика: пиксельный пробинг строки ячейки в headless-Chromium (playwright) — самый правый столбец был `(58,132,62)` вместо ожидаемого fold `(30,42,34)`. Перебор вариантов (`background-clip:padding-box`, без бордера, без radius, inner-bar) показал, что хвост убирает именно `padding-box`.

**Тупик, который сначала прошли:** первая гипотеза была про цвет тёмного fold-остатка (`#232b26` светлее фона `#1e2a22`). Это отдельная мелочь (исправлена заодно, см. ниже), но НЕ причина «хвоста» — пользователь указал что хвост **зелёный**, а не тёмный.

### Решение
1. **Главное:** на `.matrix-cell` добавлен `background-clip: padding-box !important;` (`static/style.css`). Градиент клипается внутрь бордера → зелёный край исчезает (правый пиксель стал `(23,27,24)` — бордер над фоном страницы, без зелени). `!important` обязателен: JS присваивает `cell.style.background` (shorthand), который сбрасывает `background-clip` обратно в `border-box`; stylesheet-`!important` бьёт inline-normal. Одна строка чинит все матрицы (Visualizer, Editor, hint-grid Practice — общий класс `.matrix-cell`).
2. **Заодно:** цвет fold-остатка в градиенте `#232b26` → `#1e2a22` (= фон пустой ячейки) в `visualizer.js`/`editor.js`/`drill.js`, чтобы незалитая часть сливалась с пустыми ячейками (консистентно с implicit-fold в данных).

Решение по модели отображения (после вопроса пользователю): **оставляем полоску-бар** (ширина зелёного = частота), НЕ переходим на сплошную заливку оттенком.

Cache-bust `?v=20260529k` на `style.css`/`visualizer.js`/`editor.js`/`drill.js` + index.html. Проверено в реальном приложении headless-Chromium: `computed background-clip = padding-box`, правый край всех смешанных ячеек без зелени (`green_tail=False`). `node --check` зелёный.

### Регрессии
Не обнаружено. Чистый open=1.0, пустые/fold-ячейки и середина бара — без изменений. Затронут также hint-grid в Practice (Drill) — изменение консистентно.

## P-008: В выпадашке визуализатора виден автолейбл, а не имя сохранённого диапазона

**Status:** resolved
**Found:** 2026-05-30
**Resolved:** 2026-05-30
**Severity:** medium — пользователь «не находит» свой диапазон, хотя файл сохранён и сервер его отдаёт

### Симптомы
Пользователь рисует диапазон в Editor, сохраняет под именем `NL25GTOW`, но в выпадашке Visualizer его «нет». На самом деле он есть, но показан как `Cash 6max 100bb` — совпадает с другим Cash-файлом (`cash_6max_100bb.json`, label `Cash 6max 100bb (GTO micro)`), поэтому свежий файл визуально неотличим от старого и кажется пропавшим.

### Корень
Две независимые причины, обе не про потерю данных (бэкенд `/api/ranges/list` корректно отдаёт все 9 файлов, проверено live HTTP):
1. **Editor затирает label автогенерёжкой.** `_edBuildRangeData()` в `editor.js` всегда строит `meta.label = `${game} ${table} ${depth}bb`` и игнорирует имя файла, которое ввёл пользователь. Имя `NL25GTOW` остаётся только как имя файла на диске, в label не попадает.
2. **Visualizer показывает только label.** `vizRefreshDepthOptions()` рендерил `option` как `f.label || f.stack_depth` — имя файла (`f.filename`) в UI не показывалось вообще. Автолейбл неуникален → коллизия пунктов.

### Решение
Слой показа (минимально-рисковый фикс) применён ко **всем трём** файловым выпадашкам — формат `имя_файла · label` (например `NL25GTOW · Cash 6max 100bb`). Имя файла — ведущая идентичность, мета — вторичный контекст. Edge-кейсы прикрыты: если label == имя файла или пуст, дубль не печатается (fallback на `stack_depth`; напр. `mtt_8max_15bb · 15bb`).
- `vizRefreshDepthOptions()` в `static/visualizer.js` (первый заход).
- `edPopulateLoadDropdown()` в `static/editor.js` — выпадашка «загрузить файл» (второй заход: пользователь не нашёл свой файл при дозаполнении диапазона). Тут показ ведущего имени критичнее всего — это список «что редактировать», и он не фильтруется по game/table, показывает все файлы.
- Drill file-select в `static/drill.js`.

Cache-bust в `index.html` бампнут для всех трёх: `visualizer.js`/`editor.js`/`drill.js` → `?v=20260530a` — без бампа браузер крутил бы старый JS из кэша (та же ловушка, что в P-007).

- **FOLD почти редундантен с CLEAR.** В формате отсутствующая рука = implicit fold (дополняется до 1.0), а цвет fold `#1e2a22` = фон пустой ячейки (см. `edRenderCell`, `static/editor.js:532`) → покрашенный fold рисуется неотличимо от пустой клетки. Тренажёры/оценка обрабатывают оба одинаково (implicit fold), поэтому уникальной функции у FOLD по сути нет. (Учебный контраргумент: явная кнопка «фолд» может быть интуитивнее новичку, даже если механически = clear.)
- **CLEAR шире намеренно:** `grid-column: span 2` в `static/style.css:1077`. По смыслу оправдано — это инструмент-ластик, другая категория, чем action-кнопки; визуальное отделение разумно. Сейчас подпись `✕ Clear` (крестик + текст, `index.html:114`).

### Подтверждение редундантности FOLD (проверено по коду 2026-05-30)
Модель «рука без действия = fold» верна и для полного, и для частичного фолда, и движок её уже реализует:
- **Полный:** недостающая рука → implicit fold. `srs._make_card` (стр. 152) достраивает `fold` до 1.0; сборка Learn-колоды `init_cards_from_spots(fill_implicit_fold=True)` (дефолт) разворачивает каждую недостающую из 169 рук в карточку `{fold: 1.0}`. Явный `{fold:1.0}` и отсутствующая рука дают **идентичную** карточку.
- **Частичный:** у руки `{open:0.7}` остаток `0.3` достраивается до fold автоматически (та же стр. 152). Руками дозаливать fold не нужно.
- Покрашенный FOLD рисуется неотличимо от пустой клетки (цвет `fold = #1e2a22 =` фон пустой, `edRenderCell` стр. 565); CLEAR даёт ту же пустую клетку. В Practice фолд-руки не раздаются в обоих случаях.
Единственный кейс, где явный fold отличался бы — `fill_implicit_fold=False` (legacy, колода только из явных записей), но это не дефолт и не используется в текущем Learn-потоке. Вывод: уникальной функции у FOLD нет.

### Решение (выполнено 2026-05-30)
- **FOLD убран** из бара Action — удалён `<button data-action="fold">` из `index.html` и обе CSS-правила `.paint-btn[data-action="fold"]` / `.paint-btn.active[data-action="fold"]`. Implicit fold (полный и частичный) полностью покрывает фолд; проверено по коду (см. блок «Подтверждение редундантности» выше). Grep подтвердил: ссылок `data-action="fold"` в `static/` не осталось.
- **CLEAR → иконка-корзина.** Текст `✕ Clear` заменён на 🗑 (`title`/`aria-label="Clear hand"`). Остаётся full-width снизу (`grid-column: span 2`), добавлен `font-size:16px; letter-spacing:0` под глиф.
- **Раскладка панели Action:** теперь 4 кнопки 2×2 (Open / 3-Bet / Call / 4-Bet) + широкая корзина снизу — как и просил пользователь.
- Цвет fold (`#1e2a22`) в `editor.js` COLOR-map **оставлен** — он нужен для отрисовки незалитого (implicit-fold) остатка в ячейках матрицы, к удалённой кнопке отношения не имеет.
- Отражено в `docs/design.md` (раздел «Кнопки» → Action). Cache-bust `style.css → v=20260530e` (index.html/editor.js этим фиксом не менялись по логике, только style.css + разметка).

**Не покрыто автотестом** (фронтенд без harness) — ручная проверка: панель Action показывает 5 кнопок (2×2 + корзина), клик по корзине чистит руку, покраска Open/Call/3bet/4bet работает, на споте RFI всё ок без FOLD.

### Регрессии
Не обнаружено в коде. Удаление FOLD-кнопки не трогает логику покраски (`edPaintHand` обрабатывает action обобщённо) и рендер implicit-fold остатка (COLOR-map `fold` на месте).

## P-015: Кнопка Save в редакторе не отслеживает «грязное» состояние — можно жать Save без изменений

**Status:** resolved
**Found:** 2026-05-30
**Resolved:** 2026-05-30
**Severity:** low — лишние пересохранения идентичных данных, неинтуитивно; данные не портятся

### Симптомы
Отредактировал диапазон, нажал Save → «✓ Saved!». Не сделав ни одного изменения, кнопка снова стала активной, и повторный Save сработал опять (пересохранил то же самое). Ожидание пользователя (как в любой программе): иконка дискеты активна только когда есть несохранённые правки; после сохранения — бледная/неактивная до следующего изменения.

### Корень
Нет отслеживания dirty-state. `_edDoSave` (`static/editor.js:638`) после успеха через `setTimeout` (стр. 673–679) просто возвращает кнопке `disabled = false` и дефолтный label — безусловно, без проверки «были ли изменения с момента сохранения». Понятия `ed.dirty` в редакторе нет (grep по `dirty`/`isDirty` пусто). Все точки мутации (`edPaintHand` стр. 474, `edApplyQuickSelect` стр. 203, Clear All Hands `eClearAllBtn`, изменение freq-слайдера, смена action/spot) меняют данные, но нигде не выставляют флаг «есть несохранённое».

### Решение (выполнено 2026-05-30)
Введён флаг `ed.dirty` (bool, в state-объекте `editor.js`) + три хелпера:
- `edMarkDirty()` — ставит `dirty=true` (no-op если уже true) и перегейтит UI. Вызывается из ВСЕХ точек мутации диапазона: `edPaintHand` (в начале — покрывает и одиночную покраску, и Clear отдельной руки, и `edApplyQuickSelect`, т.к. она маршрутизирует покраску групп через `edPaintHand`) и обработчик Clear All Hands. Прим.: смена частоты/action/спота/позиции данные диапазона НЕ меняет → dirty не ставит (правильно).
- `edSetClean()` — `dirty=false` + перегейт. Вызывается после успешного сохранения (внутри `if (ok)` в `_edDoSave`) и при загрузке файла (`edLoadRange` → данные = диск).
- `edUpdateDirtyUI()` — `eSaveCurrentBtn.disabled = !ed.dirty`. Гейт **только на overwrite-кнопку**; «Save as new range» (`eSaveBtn`) всегда активна (сохранение копии под новым именем осмысленно и без правок — решено как в плане).

Интеграция с транзиентным состоянием `_edDoSave`: в обоих setTimeout (success/error) после восстановления label добавлен `edUpdateDirtyUI()` — на успехе перегейтит в disabled (чисто), на ошибке оставит enabled (правки не сохранены → ещё dirty). `edUpdateSaveUI()` в конце тоже зовёт `edUpdateDirtyUI()` — при показе overwrite-кнопки она сразу в нужном состоянии. CSS: переиспользован существующий `.save-btn:disabled {opacity:.45}` + добавлен `.save-btn:disabled:hover` (не «оживать» на наведении). Cache-bust `style.css → v=20260530f`, `editor.js → v=20260530d`.

**Проверка:** `node --check` зелёный; проверена разводка вызовов grep'ом. Логика потока: загрузка → disabled (нельзя пересохранить без правок); первая покраска → enabled; overwrite-save → disabled; новая правка → enabled; save-as-new всегда активна; ошибка сохранения → остаётся enabled. **Не покрыто автотестом** (фронтенд без harness) — ручная проверка в браузере по этому сценарию.

### Регрессии
Не обнаружено в коде. Существующий pre-existing нюанс в `edSave` (строка ~774: `const ok = _edDoSave(...)` — `_edDoSave` async, `ok` = Promise = всегда truthy) НЕ трогался: сброс dirty живёт внутри `_edDoSave` (асинхронно, на реальном успехе), поэтому ложный успех в `edSave` на флаг dirty не влияет.

**Severity:** high — затрагивает КАЖДУЮ микс-руку в любом диапазоне; чем больше реальных GTO-паков с частотами, тем массовее. Подрывает доверие к тренажёру (наказывает за правильную игру).

Это **прямо противоречит принятым решениям** (`CLAUDE.md`):
- **#7** — для микс-карт любое in-strategy действие (частота > 0) = правильно.

### Решение (НЕ начато — модель оценки выбрать позднее, отложено пользователем)
Кандидаты (обсудить, выбрать до кода):

## P-019: Drill — сфолдившие/неактивные места всё равно показывают карты-рубашки

**Status:** resolved (реалистичная модель)
**Found:** 2026-05-30
**Resolved:** 2026-05-30
**Severity:** medium — вводит в заблуждение по структуре раздачи; на каждой раздаче Drill/Learn видно у всех мест карты

### Симптомы
В Drill на споте RFI (репро-скрин: Hero=MP, лог «UTG folds») у мест, которые уже сбросили, всё равно нарисованы рубашки (UTG показывает карты, хотя сфолдил). Пользователь: при смене позиции у «игроков за нами» карты есть, а должны быть сброшены. На споте vs_RFI ещё нагляднее — карты-рубашки у ВСЕХ за столом, хотя живых должно быть только двое-трое (hero + рейзер).

### Корень (подтверждён по коду)
`renderSeatsInto` (`static/drill.js:431`) рисует рубашки каждому не-hero месту безусловно. Множество сфолдивших `spec.folded` НИКОГДА не передаётся: `renderSeats` (стр. 420–429) его не формирует → `spec.folded = null` → `isFolded` всегда `false` для всех. В комментарии (стр. 440–441) это и помечено как «Engine-driven fold state is a later step» — шаг так и не сделан. Двойная недоделка: даже если задать `folded`, мини-рубашки всё равно добавляются (стр. 464–469 проверяют только `!isHero`, не `isFolded`) — нужно ещё и не дорисовывать карты сфолдившим. Фолд-множество фронт может вычислить сам из `positions/heroPos/spot/villainPos` — правка серверу не требуется (для базовой модели).

### Нюанс покерной логики (важно для выбора модели)
Строго по правилам, в RFI к моменту хода hero сфолдили только места ДО него по порядку (`positions[:heroIdx]`; их и логирует движок «{pos} folds»). Места ПОСЛЕ hero (включая блайнды) ещё не действовали → карты держат корректно. То есть «игроки за нами» (после по позиции) по покеру карты как раз НЕ сбрасывают. Поэтому фикс зависит от выбранной модели отображения:
- **Модель A — реалистичная:** RFI → гасить карты только у мест до hero (уже сфолдили); места после hero + блайнды держат рубашки. vs_RFI → живые = hero + рейзер (+ опц. ещё-не-ходившие после hero), остальные без карт.
- **Модель B — «прожектор» (упрощённая):** RFI → карты только у hero (акцент «фолдят до тебя, твоё решение»); все прочие без карт. vs_RFI → карты только у hero + рейзер.

### Решение (выполнено 2026-05-30 — реалистичная модель, выбрана пользователем)
Добавлен хелпер `foldedSetForSpot(positions, heroPos, spot, villainPos)` в `static/drill.js` (перед `renderSeatsInto`). `positions` — в порядке действий (UTG→…→BB). Возвращает Set сфолдивших:
- **RFI:** фолдят все места ДО hero (`i < heroIdx`); hero + места после (вкл. блайнды, ещё не ходили) — живые.
- **vs_RFI:** то же, но рейзер (`villainPos`, до hero) исключается из фолда → живой; места после hero — живые.
- **vs_3bet:** живы только hero + рейзер (`villainPos`, после hero); все прочие сфолдили.

`renderSeatsInto` теперь вычисляет `folded` сам из `spec` (если `spec.folded` не передан явно): `spec.folded instanceof Set ? spec.folded : foldedSetForSpot(...)`. Отрисовку карт НЕ трогали — существующий CSS уже корректно гасит карты сфолдивших: `.seat.is-folded .seat-cards { visibility:hidden }` + `.seat.is-folded { opacity:.34; grayscale }` (`style.css:360,374`). Место остаётся (токен позиции + стек, затемнённые), карт нет.

`learn.js` зовёт тот же `renderSeatsInto` и передаёт `positions/heroPos/spot(card.spot)/villainPos` → реалистичная модель применяется и в Learn автоматически, без отдельной правки. Cache-bust `drill.js → v=20260530e` (learn.js не менялся — он лишь дёргает обновлённую `renderSeatsInto`).

**Проверка:** `node --check` зелёный; `foldedSetForSpot` прогнан на 6 кейсах 6-max — все сходятся с покерной логикой: RFI·MP → фолдит только UTG (= репро-скрин); RFI·UTG → никто; RFI·SB → UTG..BTN; vs_RFI BTN<CO → фолд UTG,MP (живы CO-рейзер, BTN, SB, BB); vs_3bet UTG<BTN → живы только UTG,BTN. **Не покрыто браузерным автотестом** (нет harness в заходе) — рекомендуется ручная проверка: на RFI/vs_RFI/vs_3bet у сфолдивших мест карт нет и они затемнены, у живых — рубашки.

### Регрессии
Не обнаружено в коде. Изменение чисто аддитивное (новый хелпер + вычисление `folded`); путь отрисовки `is-folded` существовал и раньше. `spec.folded`-override сохранён для будущей engine-driven логики.
