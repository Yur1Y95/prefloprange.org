**Статус:** дизайн (2026-06-19, чат «проектируем новую ветку»). **Код не тронут.** Это
авторитетная спека ветки — её читает каждый следующий чат перед сборкой своего этапа.
Сводка-трек — `roadmap.md` Track E; архитектурное решение — `CLAUDE.md` #19; визуальная
доска — `tracks_board.md`; латентная мина — `problems.md` P-037.

## Что в ветке и чего нет

**В охвате (решение пользователя 2026-06-19):**
- Размеры столов: **heads-up (2-max), 8-max, 9-max** (сейчас только 6-max cash + 8-max MTT).
- Глубина **200bb** с расширением лестницы спотов до **vs_5bet**.

**Вне охвата (отложено сознательно, НЕ передоговариваться в этой ветке):**
- **Анте.** Цепляет короткий стек и «спины» — проектируется отдельной связкой
  (roadmap «Форматы стола — спины»). Префлоп у нас — no-ante теория (#2 в CLAUDE.md).
- **Микс размеров открытия внутри одного солва** (блок `sizing`) — отдельная отложенная
  ось (roadmap «Размеры открытий», Ось 2).
- **3-max**, полноценный тренажёр выбора размера, узлы глубже vs_5bet (6bet+).

## Главная мысль (костяк ветки)

Размер стола и глубина — уже **свойства пака**, не код. Источник истины: `meta`
(`table_size`, `stack_depth`) + `config.positions / rfi_positions / vs_*_options`.
8-max уже живёт в `gto_100bb_mtt.json` (8 позиций); heads-up — это просто пак с 2
позициями. Бóльшая часть кода читает позиции из `config` и размеро-агностична.

**Поэтому ветка — это в основном ДАННЫЕ (новые паки) + снятие нескольких 6-max-хардкодов,
а не переписывание движков.**

Прямое следствие (важное для трудозатрат): `gtow_coverage` и `gtow_capture_list` строят
список узлов **из `config.positions`**. Заведёшь скелет пака с правильным `config` —
генератор сам выдаст полный список узлов для захвата. Руками ничего не считаем.

## Три оси и что каждая ломает

**Ось 1 — размер стола (HU / 8 / 9-max).** Формат и фронт почти готовы. Реальная работа:
- **Мина №1: `POSITION_ALIASES = {"HJ":"MP"}` применяется к КАЖДОЙ прочитанной позиции.**
  В 6-max это чинит «GTOW зовёт 2-е место HJ, мы — MP». Но в 8/9-max HJ и MP — РАЗНЫЕ
  места: тот же алиас молча слепит их в одно и испортит пак. → размеро-зависимая карта (Р1).
- Фронт: `SLOT_POS[2]`/`SLOT_CHIP[2]` для HU; пресет 2max в `editor.js`.
- Канон имён позиций 8/9-max и HU — сверить на живом GTOW (Q2/Q3).

**Ось 2 — глубина 200bb.** В основном пак-метаданные + новый файл (#15). Но глубже 100bb
появляется реальный 5-бет-узел, а парсер обрезан на vs_4bet → нужен спот **vs_5bet** (Р2).
И `drill_engine.STARTING_STACK=100` (+ комментарий «6-max preflop») — параметризовать.

**Граница этапов (важно):** всё на **100bb не требует vs_5bet** — лестница open/3bet/4bet/allin
закрывает 100bb на любом размере стола. Поэтому работу по размеру стола (HU/8/9-max @100bb)
делаем БЕЗ касания обрезки парсера; vs_5bet включаем только на этапе 200bb. Не смешиваем.

## Количество узлов на пак (проверено программно 2026-06-19)

Формула: `RFI = #rfi_positions`; каждый из vs_RFI / vs_3bet / vs_4bet = число
упорядоченных пар «ранний-опенер → более позднее место» = `P`; на 200bb добавляется ещё
`P` узлов vs_5bet.

| Размер | RFI | vs_RFI | vs_3bet | vs_4bet | **Итог @100bb** | +vs_5bet | **Итог @200bb** |
|---|---|---|---|---|---|---|---|
| heads-up | 1 | 1 | 1 | 1 | **4** | +1 | **5** |
| 6-max | 5 | 15 | 15 | 15 | **50** | +15 | **65** |
| 8-max | 7 | 28 | 28 | 28 | **91** | +28 | **119** |
| 9-max | 8 | 36 | 36 | 36 | **116** | +36 | **152** |

Это число ручных кликов букмарклетом на один стейк (граница автоматизации не сдвигается —
браузером мы не управляем; см. roadmap P-020 / «полная автоматизация импорта»). Отсюда
порядок этапов: HU тривиален, 8/9-max — крупные заливки.

## Архитектурные решения

### Р1 — Алиас позиций под размер стола
`detect_spot(html, aliases=...)` принимает карту «имя GTOW → наше имя»; батч-импортёр
(`gtow_import_*`) берёт её по `meta.table_size` целевого пака. Одиночный CLI по умолчанию
6-max (`{"HJ":"MP"}`) — обратная совместимость не ломается. Карты по размеру (значения
**сверить на живом GTOW** при сборке):

| table_size | alias (GTOW → наш) | примечание |
|---|---|---|
| 6max | `{"HJ":"MP"}` | как сейчас |
| 8max | TBD (вероятно `{}` или `{"LJ":"MP"}`) | канон MTT-пака: UTG,UTG+1,MP,HJ,CO,BTN,SB,BB |
| 9max | TBD | +UTG+2 (или LJ) — уточнить |
| 2max (HU) | TBD | зависит от Q3 (SB vs BTN) |

### Р2 — vs_5bet как чисто аддитивный спот (зеркало vs_4bet)
Лестница: `open → 3bet → 4bet → (vs_5bet) → allin`. Расширяем строго по образцу уже
сделанного vs_4bet (roadmap «спот vs_4bet — done», все слои перечислены там):
- `range_engine.get_vs_5bet_range` (зеркало `get_vs_4bet_range`) + ветка в `get_ev`.
- `config.vs_5bet_options` — `gtow_autoconfig` строит из фактических данных.
- `gtow_parse`: снять cap → `nr == 4 → vs_5bet`; `RAISE_ACTION["vs_5bet"] = "allin"`.
- `gtow_coverage`: vs_5bet **present-only** (как squeeze/vs_squeeze) — НЕ требовать полную
  матрицу (не каждая 5-бет-линия стандартна), либо гейт на `stack_depth ≥ 200bb`.
- `drill_engine` / `srs` / фронт — зеркало vs_4bet.
- **Именная бородавка (честно).** vs_4bet уже зовёт агрессивное действие `allin` (на 100bb
  это и есть олл-ин). На 200bb «allin»-бакет vs_4bet фактически = «сайзовый 5бет или джем»
  (Raise+Allin схлопнуты, #13). Семантически точнее было бы `vs_4bet→5bet`, `vs_5bet→allin`,
  но это правит существующие 100bb-паки/тесты — не стоит, пока UI не потребует сайзовые
  подписи. Решаем на этапе тренажёра (ср. roadmap «кнопки vs_3bet allin/4bet/call/fold»).

### Р3 — Heads-up = вырожденный 2-позиционный пак
HU структурно проще всех: 2 места, 4 узла. RFI = SB (=кнопка, ходит первой префлоп)
открывает; vs_RFI = BB отвечает; vs_3bet = SB против 3-бета BB; vs_4bet = BB против 4-бета
SB. Спот-детект (0/1/2/3 рейзера) ложится без правок. Именно поэтому HU — лучший первый
этап: прогоняет весь путь «новый тип стола» end-to-end при минимальном риске.

## Лестница этапов (каждый — отдельный чат, design-first, тесты в «готово»)

**E0 — дизайн** *(done 2026-06-19, этот чат)*. Спека (этот файл) + Track E + CLAUDE #19 +
P-037 + доска.

**E1 — Heads-up @ 100bb.** Самый дешёвый, доказывает плумбинг нового типа стола.
- Сверить на живом GTOW: есть ли HU-cash-солв 100bb; имена позиций; сайзы.
- Скелет пака `data/<HU>.json` (`meta.table_size="2max"`, `config.positions` per GTOW).
- Парсер: HU-запись в алиас-карту (Р1); подтвердить, что detect_spot отдаёт RFI/vs_RFI/…
- Захват 4 узлов (букмарклет) → импорт.
- Фронт: `SLOT_POS[2]`/`SLOT_CHIP[2]`, пресет 2max в editor.
- **Готово:** пак грузится через `range_engine`; coverage 4/4; Drill/Learn рисуют 2 места;
  существующие тесты зелёные.

**E2 — Алиас под размер стола (рефактор парсера).** Снять мину №1 (Р1). Фундамент для 8/9.
- **Готово:** `tools/test_gtow_tools.py`/`test_gtow_parse.py` + новый кейс: 8-max-дамп
  оставляет HJ как HJ, 6-max-дамп всё ещё HJ→MP. Регрессий нет.

**E3 — 8-max @ 100bb.** Первый широкий пак (~91 узел).
- Сверить на живом GTOW имена/сайзы 8-max → финализировать 8-max-алиас и канон позиций.
- Скелет (8-max config) → `gtow_capture_list` выдаст узлы → захват → импорт. `SLOT_POS[8]` есть.
- **Готово:** coverage 7/28/28/28; пак грузится; тесты зелёные.

**E4 — 9-max @ 100bb.** То же, ~116 узлов; `SLOT_POS[9]` есть. **Готово:** coverage 8/36/36/36.

**E5 — 200bb + лестница vs_5bet.** Только теперь трогаем обрезку парсера (Р2).
- vs_5bet во всех слоях (Р2); `drill_engine.STARTING_STACK` из `meta.stack_depth`.
- Сверить 200bb-сайзы на живом GTOW; завести 200bb-пак (выбрать размер стола), захватить
  (узлов больше — добавился vs_5bet).
- **Готово:** vs_5bet round-trip через `range_engine`; coverage учитывает vs_5bet
  (present-only); пак грузится; тесты зелёные (включая новый vs_5bet-кейс парсера).

- `equity*.py` / постфлоп — заморожены.
