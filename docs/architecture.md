## 1. Главный вывод

**«Слишком много файлов» — это про дерево, а не про код.** Цифры на 2026-06-17:

- Всего файлов в рабочей папке: **857**. В git отслеживается: **188**. Остальные ~670 — локальный
  «не-код»: база рук, бэкапы, дампы GTOW, личные заметки (всё под `.gitignore`).
- Собственно **рантайм-кода ≈ 13 Python-модулей + 7 JS-файлов** (~3,5к строк Python с тестами,
  ~4,4к JS). Это маленький, аккуратный проект.

Сложность проекта измеряется **графом зависимостей кода**, а не числом файлов. Поэтому план —
не «переписать архитектуру» (она здоровая), а **навести гигиену дерева**: сгруппировать файлы по
роли, изолировать замороженное, выкинуть мусор.

## 2. Слои и направление зависимостей

Зависимости идут строго в одну сторону (верхние слои знают про нижние, не наоборот) — это и есть
признак здоровой слоёной архитектуры:

```
Браузер  →  static/index.html + 7 JS-модулей
                      │  HTTP /api/*
                      ▼
Composition root  →  main.py  (собирает роутеры, отдаёт UI, эндпоинты drill/ranges/stats/history)
                      │  include_router()
                      ▼
API-роутеры       →  srs_api │ equity_api (frozen) │ postflop_api (frozen)
                      │  вызывает
                      ▼
Доменное ядро     →  range_engine + drill_engine │ srs │ equity → decision → postflop_engine (frozen)
                      │  читает
                      ▼
Данные            →  data/*.json · srs_state/ · stats.json · history.json   (БД нет)
```

**Ядро = модули с нулём проектных импортов:** `range_engine.py`, `srs.py`, `equity.py`. Они ни от
чего внутри проекта не зависят, покрыты тестами, и менять их надо осторожно (особенно `equity.py` —
трижды сверен с equity.poker). Всё остальное зависит от них, а не наоборот.

## 4. Как думать об архитектуре (4 модели для навигации)

1. **Отделяй код от не-кода.** Когда проект «ощущается огромным» — посчитай строки *исполняемого
   кода*, а не файлы. Тут кода мало; огромным дерево делают данные, бэкапы и заметки.
2. **Слои и направление стрелок.** Главный вопрос: «кто от кого зависит, и однонаправленны ли
   зависимости?» (см. §2). Привычка рисовать этот граф мгновенно «уменьшает» любой проект в голове.
3. **Горячее vs холодное.** `equity.py` — холодное ядро (проверено, не трогаем). `drill.js` —
   горячее (меняется еженедельно). Знание, что есть что, говорит, где осторожничать.
4. **«Один модуль — одна задача».** Уже соблюдается — именно поэтому код остался чистым, пока дерево
   разрасталось. Это главная защита, держимся её. Новый функционал → новый файл с роутером → одна
   строка `include_router` в `main.py`.

## 5. Что упростить (без переписывания логики)

1. **Сгруппировать файлы по роли.** Самый большой выигрыш при нулевом риске. Минимум: тесты в
   `tests/` (в `CLAUDE.md` дерево уже рисует `tests/`, но по факту все `test_*.py` лежат в корне —
   это дрейф документации), рантайм-модули — в `app/`. Перенос Python меняет импорты и `Dockerfile`,
   поэтому делать **отдельной подзадачей с прогоном тестов**, а не между делом.
2. **Изолировать замороженное.** Постфлоп (`equity`/`decision`/`postflop_*`/`hand_classify`/
   `postflop.js`) и FSRS-эксперимент (`srs_fsrs` + `tools/fsrs_*`) спят, но раскиданы среди живых
   файлов. Собрать в `postflop/` и `fsrs_experiment/` — открыл проект и сразу видно, что живо.
3. **Перестать делать бэкапы руками.** 5 папок снапшотов колоды + ~30 `.bak` в `data/` (у `NL25GTOW`
   — 25 копий) — это страх перед git. Git и есть бэкап и машина времени (`git log`, `git checkout
   <файл>`). Освоить — и руками снапшоты не нужны.
4. **Подрезать `CLAUDE.md`.** Он грузится в каждую сессию, а распух (решение #13 — десяток абзацев,
   по абзацу на каждый импортированный GTOW-пак). Принципы оставить в `CLAUDE.md`, журнал импортов
   пакетов увести в `roadmap.md`. Длинный always-on файл и дорогой, и быстрее дрейфует.
5. **Подрезать дампы GTOW.** 289 html-дампов (под `.gitignore`, но раздувают дерево). Одна папка на
   лимит, чистить после импорта.

## 7. Материалы по архитектуре и структуре кода (по порядку полезности)

1. **The Grug Brained Developer** — grugbrain.dev. Бесплатно, коротко, смешно. Про то, что сложность
   — главный враг и не надо переусложнять. Читать первым.
2. **«A Philosophy of Software Design», John Ousterhout** — тонкая книга для начинающих про модули,
   «глубокие» интерфейсы и где прячется сложность. Лучшее «как структурировать код» для твоего уровня.
3. **FastAPI → «Bigger Applications - Multiple Files»** (офиц. дока) — прямо твой стек: ты уже
   используешь роутеры, тут показано, как разложить FastAPI-приложение по папкам.
4. **Real Python → «Python Application Layouts»** — конкретные конвенции структуры (`app/`, `tests/`).
5. **refactoring.guru** — бесплатный визуальный каталог «запахов кода» и рефакторингов.
6. **Cosmic Python** — cosmicpython.com (бесплатная книга, слоёная архитектура). Уровень 2, в закладки.
