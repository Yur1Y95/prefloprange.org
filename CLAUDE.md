## О проекте

Веб-тренажёр покерных диапазонов и постфлоп-решений. Универсальный — не привязан к конкретному руму или формату, работает с любыми диапазонами в нашем JSON-формате. Целевые игры: **NL Hold'em**, **6-max кэш** и **MTT 8-max**, глубина 100bb.

Конечная цель — собственный инструмент для тренировки и анализа: визуализатор + редактор диапазонов, тренажёр префлопа и постфлопа, в перспективе анализатор сыгранных раздач.

## О пользователе

- **Уровень в программировании:** начинающий. Проект — способ одновременно подтянуть код и выбраться с микролимитов в покере. Объясняй технические решения, не вали в код без пояснений.
- **Язык общения:** русский. Код, имена файлов, переменные — английский (это явно принятое решение).
- **Стиль работы:** обсуждаем дизайн перед написанием кода. Большие шаги декомпозируем на этапы с проверкой. Не делаем нескольких рискованных вещей одновременно.
- **OS:** macOS. Проект лежит в `~/Desktop/Poker_Trainer/`.

## Технологический стек

- **Backend:** Python 3, FastAPI. Файлы запускаются как обычно через `uvicorn main:app --reload` или просто `python3 main.py`.
- **Frontend:** чистый HTML + CSS + JS, без фреймворков и без сборщика. Подаётся как статика из `static/` через FastAPI.
- **Persistence:** JSON-файлы в проекте (`stats.json`, `history.json`, `srs_state/*.json`). Никакой БД.
- **Диапазоны:** JSON в `data/`. Один файл — один range pack (cash 100bb 6-max, MTT 100bb 8-max и т.д.). Никакого хардкода диапазонов в коде.

## Формат range-файлов (JSON)

```jsonc
{
  "meta": {
    "game_type":    "MTT",            // "Cash" | "MTT"
    "table_size":   "8max",
    "stack_depth":  "100bb",
    "label":        "GTO 100bb MTT 8-max"
  },
  "config": {
    "positions":       ["UTG", "UTG+1", "MP", "HJ", "CO", "BTN", "SB", "BB"],
    "rfi_positions":   ["UTG", "UTG+1", "MP", "HJ", "CO", "BTN", "SB"],
    "vs_rfi_options":  { "BTN": ["UTG", "MP", "CO"], ... },
    "vs_3bet_options": { "UTG": ["BTN", "SB", "BB"], ... },
    "squeeze_options": { "BB": ["BTN-SB", "CO-BTN"], ... }   // hero: ["<opener>-<caller>", ...]
  },
  "spots": {
    "RFI": {
      "UTG": { "AA": {"open": 1.0}, "A5s": {"open": 0.7}, ... }
      // missing hands = implicit fold (skipped in trainers/SRS)
    },
    "vs_RFI": {
      "SB": {
        "vs_UTG": { "AA": {"3bet": 1.0}, "AKs": {"3bet": 0.6, "call": 0.4}, ... }
      }
    },
    "vs_3bet": { ... },
    "squeeze": {
      "BB": {
        "BTN-SB": { "AKs": {"squeeze": 1.0}, "99": {"squeeze": 0.6, "call": 0.4}, ... }
        // sub-key "<opener>-<caller>" — порядок посадки: опенер первым, коллер вторым
        // разделитель "-" (НЕ "+": позиция может содержать "+", напр. UTG+1 → split сломался бы)
        // squeeze = ровно 1 рейзер (опенер) + >=1 коллер + hero; мульти-коллер позже: "opener-c1-c2"
      }
    }
  }
}
```

**Имена действий зависят от спота:** RFI → `open`, vs_RFI → `call` / `3bet`, squeeze → `call` / `squeeze`, vs_3bet → `call` / `4bet`, vs_4bet → `call` / `allin`, iso → `raise` / `call`. Implicit `fold` дополняется до 1.0 только если total < 1.0. Не выдумывай новые имена действий — фронтенд и оценка завязаны на эти.

Старый ключ `ranges` поддерживается через `range_engine._normalize_range_data` (зеркалится в `spots`). Не вычищай — есть legacy файлы.

**Опциональный блок `ev` (GTO-EV прибыльного действия).** Параллелен `spots`, держится ОТДЕЛЬНО от него — внутрь action-словаря EV не кладём (иначе его подхватят `available_actions`, `calculate_rfi_stats`, `expand_rfi_range`, матрица-подсказка). Структура зеркалит `spots`, значения — EV в bb, per-hand (точные ключи, без нотации):

```jsonc
"ev": {
  "RFI":    { "UTG": { "AA": 2.31, "A5s": 0.42 } },
  "vs_RFI": { "SB":  { "vs_UTG": { "AKs": 1.2 } } },   // на будущее
  "squeeze":{ "BB":  { "BTN-SB": { "AKs": 1.83 } } }   // зеркалит spots.squeeze
}
```

Лукап — `range_engine.get_ev(data, spot, position, hand, villain_position=None)` → `float | None`. `None` = нет блока / нет записи → **EV не показываем** (тренажёр падает на чистое правильно/неправильно). Сейчас заполнен только `RFI`. Показ в Drill (решение #12 ниже): EV только на правильном открытии; правильный фолд и любая ошибка — без EV.

## SRS Module — Технический справочник

**Что это.** Spaced Repetition System для тренировки запоминания диапазонов в Learn Mode. Реализует SM-2 алгоритм в стиле Chessable: объективная оценка (попал в стратегию / не попал) + опциональная кнопка «Легко» для известных карточек. Логика и почему выбран этот подход см. в разделе «Ключевые решения» #6–#9.

**Файлы:**
- `srs.py` — модель `Card`, SM-2 алгоритм, persistence. Без зависимостей кроме stdlib.
- `srs_api.py` — FastAPI router, эндпоинты `/api/srs/*`
- `srs_state/<range_stem>.srs.json` — состояние deck per range file
- `test_srs.py` — 64 юнит-теста
- `test_srs_api.py` — 18 интеграционных тестов против реального range-файла

### Модель Card

Одна карточка = одна (рука, hero позиция, спот, villain позиция). Пример: `AKs SB vs_RFI vs UTG`.

```python
@dataclass
class Card:
    # Identity (неизменно за жизнь карточки)
    hand: str                       # "AKs", "T9o", "22"
    position: str                   # hero: "UTG", "MP", "CO", "BTN", "SB"
    spot: str                       # "RFI" | "vs_RFI" | "squeeze" | "vs_3bet"
    villain_position: str = ""      # пустое для RFI; squeeze: пара "<opener>-<caller>"
    correct_strategy: dict[str, float]  # {"open": 0.7, "fold": 0.3}

    # SRS state
    ease_factor: float = 2.5        # стартовая лёгкость
    interval_days: int = 0          # 0 = new, never reviewed
    next_review: str = ""           # ISO date
    last_seen: str = ""

    # Stats
    consecutive_correct: int = 0
    total_seen: int = 0
    total_correct: int = 0
```

`card.card_id` — детерминированный ID для API: `"AA__UTG__RFI"` для RFI или `"AKs__SB__vs_RFI__UTG"` для vs-спотов. Squeeze ложится на vs-форму, villain = пара: `"AKs__BB__squeeze__BTN-SB"` (поля по-прежнему режутся по `__`; внутри пары разделитель `-`, а не `+` — позиция может содержать `+`, напр. `UTG+1`).

### Имена действий по спотам

| Спот | Возможные действия | Пример карточки |
|---|---|---|
| RFI | `open`, `fold` (implicit) | `{"open": 1.0}` |
| vs_RFI | `call`, `3bet`, `fold` | `{"call": 0.6, "3bet": 0.4}` |
| squeeze | `squeeze`, `call`, `fold` | `{"squeeze": 0.6, "call": 0.4}` |
| vs_3bet | `call`, `4bet`, `fold` | `{"call": 0.3, "4bet": 0.7}` |
| vs_4bet | `call`, `allin`, `fold` | `{"allin": 0.8, "call": 0.2}` |
| iso | `raise`, `call`, `fold` | `{"raise": 0.7, "call": 0.3}` |

Implicit `fold` дополняется до 1.0 только когда сумма частот в файле < 1.0. UI-кнопки строятся по типу спота, а не по ключам в `correct_strategy` — иначе для pure-карточек одна из кнопок пропадёт.

### SM-2 алгоритм — поведение по рейтингам

| Рейтинг | Значение | `interval_days` | `ease_factor` | Когда вызывается |
|---|---|---|---|---|
| AGAIN = 1 | Снова | сброс в 1 | −0.20 (минимум 1.3) | Действие НЕ в стратегии |
| HARD = 2 | Тяжело | ×1.2, минимум +1 день | −0.15 | Не используется в Learn UI |
| GOOD = 3 | Хорошо | по graduation, затем ×ease | без изменений | Действие в стратегии, без Easy-флага |
| EASY = 4 | Легко | ×ease × 1.3 | +0.15 | Действие в стратегии + пользователь нажал «Легко» |

**Graduation ladder для GOOD:** `0 → 1 → 3 → 3×ease → ...`. То есть впервые увидел → завтра, второй раз правильно → через 3 дня, третий → через 7-8 дней, дальше геометрически по ease.

**«Learned» = `interval_days >= 21`** — порог попадания в долговременную память (для дашборда).

**Константы в `srs.py`:**
```python
PURE_THRESHOLD = 0.95          # частота выше этого → стратегия pure_<action>
DEFAULT_EASE = 2.5             # стартовая лёгкость новой карточки
MIN_EASE = 1.3                 # пол ease после многих ошибок
NEW_CARDS_PER_DAY = 15         # лимит новых карточек в очереди (Anki-default)
LEARNED_THRESHOLD_DAYS = 21    # порог «долговременной памяти»
```

**Порядок ввода новых карт + лимит (поправка 2026-06-15, P-029).** `get_due_cards` отдаёт новые карты НЕ в порядке колоды (= порядок рук в JSON, пары/топ-suited первыми), а в **стабильном шафле по `md5(card_id)`** — детерминированно, не зависит от JSON-порядка, стабильно между днями. Иначе дневной лимит всегда срезал один и тот же верх диапазона и хоронил маргинальные руки границы. `NEW_CARDS_PER_DAY=15` остаётся дефолтом, но теперь **настраивается через API**: `/api/srs/next` и `/api/srs/answer` принимают `new_limit` (фронт-селектор `10/15/30/50/∞`, `∞`→100000). Лимит — кап показа очереди, НЕ свойство колоды (в `srs_state` не пишется, после init не лочится в отличие от scope).

### grade_answer — мост UI → SM-2

```python
srs.grade_answer(card, user_action: str, marked_easy: bool = False) -> int
```

Логика:
- `user_action` есть в `correct_strategy` с частотой > 0 → `GOOD` (или `EASY` если `marked_easy=True`)
- `user_action` отсутствует или с частотой 0 → `AGAIN` (даже если флаг Easy выставлен — флаг игнорируется на неправильном ответе)

**Микс-карточки.** Для `A5s UTG = {"open": 0.7, "fold": 0.3}` ОБА действия (open и fold) грейдятся как GOOD. Это сознательное решение (см. #7 в «Ключевых решениях»): знание про микс-природу спота важнее совпадения с ГСЧ-розыгрышем.

### API endpoints

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/srs/status?file=...` | Инициализирован deck? Counts (total/new/due_today/young/learned) |
| POST | `/api/srs/init` | Body: `{file, scope?, force?}`. Создать deck из range-файла. 409 если deck уже есть и `force` не выставлен |
| GET | `/api/srs/next?file=...` | Следующая карточка БЕЗ `correct_strategy` в payload (не спойлерим ответ). Опц. `&new_limit=N` — кап новых карт/день (дефолт `NEW_CARDS_PER_DAY`) |
| POST | `/api/srs/answer` | Body: `{file, card_id, user_action?, marked_easy?, reveal?, new_limit?}`. `reveal=true` («Показать ответ») → грейд AGAIN независимо от action, `grading.revealed=true`. `new_limit?` — кап новых карт/день для очереди следующей карточки. Вернёт `{grading, card (с раскрытой стратегией), next, queue_size}` |
| GET | `/api/srs/summary?file=...` | Только counts (то же что в status, но без поля `initialized`) |
| POST | `/api/srs/reset?file=...` | Стереть SRS-state файл для этого диапазона |

**Параметр `scope` для init:** ограничивает типы спотов в deck. Например `["RFI"]` создаст deck только из RFI-карточек (для MVP/Phase 1). По умолчанию — все три спота.

### Типичный flow для фронтенда

```
1. GET  /api/srs/status?file=...
   → если initialized=false, показать кнопку "Начать обучение"
2. POST /api/srs/init  (один раз)
3. GET  /api/srs/next?file=...
   → показать карточку, кнопки действий по типу спота + «Показать ответ»
4. POST /api/srs/answer
   → action-кнопка: {user_action}; «Показать ответ»: {reveal:true}
   → показать grading (correct / wrong / revealed), раскрыть correct_strategy,
     показать кнопку "Дальше"
5. Повторять 3-4 до queue_size=0 на сегодня
```

## Ключевые решения — НЕ передоговариваться без явной причины

1. **Универсальность.** Тренажёр не привязан к конкретному руму или формату. Работает с любым диапазоном в нашем JSON-формате.

2. **Префлоп — теоретические (no-ante) диапазоны.** Не выводим их из базы рук. Берём из солверов/готовых ренджей.

3. **Постфлоп — диапазоны оппонентов из реальной базы.** Используем 3 млн рук с CoinPoker (ante-game) с допущением, что постфлоп-поведение типа игрока слабо зависит от ширины префлопа. Это не идеально, но рабочее приближение.

4. **Терн/ривер — сквозная раздача с сужением, не независимые улицы.** Содержательно сложнее, но честнее.

5. **Equity-движок свой (`equity.py`), не внешний.** Сверен с equity.poker, никаких внешних солверов не подключаем — это лишняя зависимость.

6. **SRS — Chessable/Duolingo-стиль, не Anki-стиль.** Объективная бинарная оценка (попал в стратегию → Good, не попал → Again). Кнопки Hard и Again-как-самооценка убраны: ввод у нас принудительный (клик действия), система объективно знает результат, заставлять пользователя самооценивать поверх — лишний шаг. Это решение зафиксировано после обсуждения с обоснованием — переоткрывать без серьёзной причины.

    **Поправка от 2026-06-01 (по решению пользователя): кнопка Easy убрана из UI, добавлена третья кнопка «Показать ответ».** Причина: принудительный бинарный ввод (open/fold) ломает сигнал SRS на угадывании — на споте с единственным верным действием случайный клик даёт 50% «правильно», карточка получает GOOD и растягивает интервал, хотя её не выучили. Третья кнопка «Показать ответ» — честный выход из лотереи: пользователь, который не знает ответа, жмёт её вместо того чтобы угадывать. Грейдинг: open/fold в стратегии → GOOD; open/fold мимо → AGAIN; «Показать ответ» → раскрытие + AGAIN (как незнание). «Не знаю» и «ошибся» для алгоритма пока одинаковы (оба AGAIN) — развести можно на A.4. Название именно «Показать ответ» (не «не уверен»): описывает действие/состояние памяти, а не эмоцию — «не уверен» провоцирует тревожную перестраховку и сбивает интервалы. Кнопка равного веса с action-кнопками (не стыдим честность), но нейтрального (не-стратегического) цвета `btn-reveal`. Easy убрана только из UI: backend-примитив EASY-рейтинга, `srs.upgrade_good_to_easy` и эндпоинт `/api/srs/upgrade_easy` оставлены нетронутыми (часть проверенного SM-2-ядра, 81 тест зелёный), просто больше не вызываются из фронта. Реализация: `srs_api.AnswerRequest.reveal: bool`, ветка reveal→AGAIN в `/answer` (+`grading.revealed`); `static/learn.js` (`?v=20260601a`) кнопка + `submitAnswer`; `static/style.css` (`?v=20260601a`) `.btn-reveal` + `.lfb-verdict.revealed`. Тест `test_srs_api.test_answer_reveal_forces_again`.

7. **SRS «правильно» для микс-карт = любое in-strategy действие.** На карточке `A5s UTG = {open: 0.7, fold: 0.3}` и open, и fold оцениваются как GOOD. Это знание про микс-природу спота, а не угадывание ГСЧ.

8. **Два режима тренажёра — Practice и Learn — это разные системы.** Learn использует SRS. Они не пересекаются, статистика отдельная.

    **Поправка от 2026-05-31 (по решению пользователя): Practice оценивает по членству в стратегии, а НЕ по броску ГСЧ.** Любое действие с частотой > 0 (включая implicit fold) считается верным — так же, как в Learn (#7). Для `66 = {open: 0.27}` и open, и fold верны; для `AA = {open: 1.0}` fold неверен (implicit fold нет, сумма = 1.0). Раньше «верность» определял бросок ГСЧ (`resolve_mixed_action`) → открытие микс-руки часто помечалось «неверно», хотя рука в диапазоне — это и был баг P-018. **ГСЧ-розыгрыш остаётся только как иллюстрация** («бросок сказал raise, в следующий раз может fold — не привязывайся»), он больше не решает правильно/неправильно. 15. **Глубины стека и размеры открытий — отложены, путь добавления зафиксирован (2026-06-14).** В GTOW выбираются и глубина (100/200/300bb), и размер открытия (2/2.25/2.5/3bb/GTO-сайз). Сейчас НЕ реализуем ни то, ни другое — у одного игрока на микрах ценность точного сайза и нестандартных глубин низкая, а цена реализации высокая. Принцип: не платим сложностью за функцию, которой ещё не пользуемся, но оставляем чистый аддитивный путь. **Две независимые оси, решаются по-разному:**

    **Ось 1 — глубина стека.** Уже заложена на уровне `meta.stack_depth`: глубина = свойство всего пака, один файл = одна глубина (`cash_100bb_6max.json`, `cash_200bb_6max.json`, …). Добавление новой глубины — это новый range-файл, **ноль правок движков** (SRS-стейт и статистика у каждого пака свои по `<stem>`). Пока остаёмся на 100bb; заводим новый пак в день, когда реально тренируем другую глубину. Заранее не закладываемся.

    **Ось 2 — размер открытия.** Различать два случая. **(A) Разные солвы под фиксированный сайз** (GTOW даёт цельную стратегию под «2.5bb», другую под «3bb») — решается как ось 1: сайз уходит в `meta`/имя пака, без правок движков. **(B) Микс размеров ВНУТРИ одного солва** (одна рука миксует open 2.5bb 60% + open 5bb 25% + fold 15%) — наш формат это сейчас ТЕРЯЕТ: парсер по конвенции #13 сливает все красные сегменты (Raise+Allin) в один бакет `open`, размер не пишется (имена действий плоские: `open`/`3bet`/`4bet`).

    **Когда понадобится случай B — реализуем по аналогии с блоком `ev` (#13), это решённый дешёвый паттерн.** Добавляем отдельный top-level блок **`sizing`**, параллельный `spots` (НЕ внутрь action-словаря — иначе подхватят `available_actions`/грейдинг, как и с `ev`), зеркалящий структуру `spots`; значения — распределение размеров per-hand: `sizing.RFI.UTG.AKs = {"2.5": 0.6, "5.0": 0.25}` (fold вне sizing). Лукап `range_engine.get_sizing(...)` → `dict | None`; `None` → размер не показываем. Грейдинг остаётся **бинарным** («рука в диапазоне открытия → верно»), drill/SRS блок игнорят — `sizing` только для показа в UI («GTO открывает 2.5bb, иногда 5bb»). Парсер дописать реально: размеры в DOM GTOW есть (кнопки сайзов + оттенки красного на сегментах ячейки), сейчас намеренно схлопываются — надо перестать схлопывать и писать в `sizing`. **Полный тренажёр выбора размера** (сайз = действие, drill/SRS заставляют выбрать правильный размер, кнопки/card_id/грейдинг по размеру) — это уже подход поверх `sizing`, трогает весь стек (формат, range_engine, drill_engine, srs, фронт, парсер); делаем только когда диапазоны открытия уже уверенно выучены и осталось докрутить «чем». Не начинать без явного решения. Подробности — `docs/roadmap.md` → отложенная задача «Размеры открытий / глубины стека».

- **Будь честен про ограничения.** Если не уверен — скажи. Если что-то сложнее чем кажется — скажи. Если решение не оптимальное — назови это, не маскируй.
- **Тестируй то, что пишешь.** Особенно математику (equity, SM-2). Не сдавай код без прогона тестов.
- **Объясняй.** Пользователь учится — пара предложений «почему так» важнее краткости.

## Как вести работу по чатам (Cowork)

Модель работы: **один чат — одна дискретная задача/подзадача.** Между чатами история разговора НЕ переносится — переносятся только файлы на диске. Поэтому документы ниже это механизм передачи состояния между чатами, а не бюрократия.

**Карта документов (что где искать):**
- `CLAUDE.md` (этот файл) — архитектура, принятые решения, конвенции. Автозагружается в каждый чат.
- `docs/roadmap.md` — четыре параллельных трека (A/B/C/D), их статус и приоритеты. **Кросс-чатовое отслеживание задач живёт здесь**, не в task-листе (task-лист эфемерный, живёт только внутри одного чата).
- `docs/problems.md` — журнал багов: проблема → решение → регрессии. Записи P-NNN.
- `docs/design.md` — дизайн-система: направление (GG-стол + GTOW-строгость), токены, инвентарь компонентов.

**Начало чата.** Пользователь называет задачу и трек одной строкой (например «Track D.1.1 — поднять Postgres + seed»). Claude первым делом читает релевантные доки: нужный трек в `roadmap.md`, связанные баги в `problems.md`, для визуальных задач — `design.md`. `CLAUDE.md` уже загружен.

**Конец чата (definition of done).** Обновить документы под реальность: статус подзадачи в `roadmap.md`, новые/закрытые баги в `problems.md`, новые архитектурные решения здесь в `CLAUDE.md`. Без этого следующий чат не узнает что произошло. Claude делает это сам в конце задачи.

**Когда новый чат, когда тот же.** Новый — на новый трек или крупную подзадачу, либо когда текущий чат стал длинным а задача закрыта. Тот же — для тесно связанных шагов в одну посадку и быстрых follow-up правок к только что сделанному.

**Раскладка треков на чаты:** Track B — каждая подзадача (фишки / анимации / матрица / мобилка) свой чат, не смешивать. Track D — каждый шаг D.1.x свой чат, они крупные. Track A.2 (практика) — без чата. Track C — мелочь, пристёгивается хвостом.
